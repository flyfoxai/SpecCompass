# Findings

- The current working-tree prompt contains explicit Level 1 portfolio decomposition, capability ownership, independent outcome, business handoff, and no-split-quota requirements.
- Prompt-contract string tests prove the wording is present, but do not prove a model output satisfies the semantic contract.
- The previously observed candidates combine multiple responsibilities, especially `数据、配置与风险边界`, and therefore require live semantic testing.
- With the safely transported current prompt, Claude produced six focused candidates: fact capture, strategy intent, risk decision, order execution, interruption recovery, and parameter governance.
- Gemini improved to four business candidates but combined normal transaction-fact confirmation with interruption recovery inside one declared chain.
- Root cause: the current validator proves one project references one chain, but it does not prove that the chain itself owns one trigger class, one business state, and one independently accepted result. A model can therefore define an impure chain first and satisfy all later cardinality checks.
- The repair needs structured chain semantics and deterministic Level 1 purity checks; adding more generic anti-boilerplate phrases alone cannot close this loophole.
- Re-running the current modified contract improved Gemini from four candidates to six: fact capture, strategy intention, risk decision, order execution, interruption reconciliation, and configuration governance. Gemini no longer merged normal fact processing with recovery.
- Claude still produced only three candidates. It separated recovery and governance, but collapsed five independently observable primary responsibilities (fact capture, strategy intention, risk decision, order submission, and broker lifecycle fact update) into one `Live-trading core loop` chain.
- This reveals a second bypass: a model can keep one trigger class and one terminal primary outcome while demoting intermediate independently accepted ownership checkpoints to steps inside a broad end-to-end chain.
- The current schema cannot deterministically expose this because `capability_atom` lacks explicit owned-state and primary-outcome identity, and Level 1 allows multiple atoms to belong to one candidate chain.
- A line-numbered recheck of the outline-discovery schema shows one `downstream_handoff` property only. The earlier duplicate-key observation was stale or came from truncated output and is not a current defect.
- Approved repair: initial Level 1 Discovery is atom-first. One source-backed responsibility atom owns one trigger/input, one state, one independently accepted result, and one handoff; it maps one-to-one to one business chain and one candidate project. Merging remains a user-visible decision option and is never performed during initial generation.
- The new regression topology is valid: every invalid fixture passes schema and the previous semantic validator, so the missing protection is specifically deterministic one-to-one ownership and consistency validation rather than malformed test data.
- Both CLI and browser validators already index capability atoms, business chains, Level 1 project nodes, and question candidates. Their current checks require non-empty same-chain references but do not require exact one-to-one cardinality or exact atom/chain semantic equality, so the fix can tighten existing validation paths without adding a parallel mechanism.
- The remaining focused methodology-test failure was a documentation contract mismatch only: the maintained methodology described how to classify transactional consistency and bidirectional handoffs but omitted the explicit phrase that they "must be classified first". The installed Codex command contract already passed.
- After fixing that sentence, the only remaining full-methodology failure was another documentation vocabulary mismatch: `docs/reference/sp-project-methodology.md` described the right diagram type as `候选子项目分图` but omitted the canonical term `业务分图` required across maintained templates.
- The final atom-first live rerun produced the same six-way ownership split in both models. Gemini returned market/broker facts, strategy intention, risk decision, order execution, state reconciliation, and parameter governance. Claude returned market/account facts, trade intention, risk gate, order lifecycle, state reconciliation, and configuration governance.
- Mechanical review of both final outputs gives `capability atom count = business chain count = candidate project count = 6`; every candidate owns exactly one atom and one chain, recovery and governance remain separate, and neither output contains a broad live-trading-core candidate.
- The live-evaluation fixture still said a candidate could contain `one or more atom labels`. The production `/sp.prd` contract overrode that weaker phrase during the successful rerun, but the fixture itself was ambiguous and is now aligned to exactly one atom per candidate.
- Claude's focused patch review found no reproducible Critical or Important defect and judged the CLI/browser semantic contract equivalent.
- Gemini's focused patch review reported two supposed Critical test/documentation failures. Both are non-reproducible: the full methodology suite already passes, and the allegedly missing maintained documents are modified in the full worktree but were omitted from the compact review bundle. No code change is justified by those findings.
