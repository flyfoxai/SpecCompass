# PRD Outline 思维导图调整研讨

## 目标

在不破坏现有 PRD Outline 成熟度、来源追踪和正式确认门禁的前提下，形成以 XMind 式项目全局思维导图为主视图、以分支为决策上下文的调整方案。

## 实施阶段

- [x] 阶段 1：以 TDD 增加导图拓扑、稳定节点绑定和密度预算 Schema，并同步 CLI、浏览器和写回校验器。
- [ ] 阶段 2：把 Outline Discovery 主视图改成全局总图、业务分图和全局约束图，右侧面板只显示当前分支的问题与自由输入。
- [ ] 阶段 3：更新 `/sp.prd`、README 和中英文方法文档，明确生成、拆图、刷新模板和非授权边界。
- [ ] 阶段 4：完成模板分发、浏览器、写回和完整回归验证。
- [x] 阶段 5：审查并设计 Level 1 业务能力生成机制，明确 Constitution/PRD 职责边界、业务语义合同与质量门禁。
- [ ] 阶段 6：以 TDD 升级 Discovery v3 业务语义合同、确定性校验和 Constitution 只读快照。
- [ ] 阶段 7：更新 `/sp.prd` 两阶段生成机制、成熟度门禁和维护文档。
- [ ] 阶段 8：调整 renderer 的业务优先展示和 Constitution 只读区域。
- [ ] 阶段 9：完成模板分发、端到端样例和三模型最终审核。

每个阶段结束后，分别调用本地 Gemini、Claude 审核，再由 Codex 对照代码、测试和用户要求裁决；Critical/Important 问题修复后才能进入下一阶段。

## 固定约束

- 一级、二级仍需用户深度参与，三级仍由 Constitution 约束模型补齐。
- 用户选择必须绑定导图节点或全局政策/治理议题，不能脱离项目上下文成为孤立问卷。
- Discovery 不能授权 `/sp.specify`；三级正式确认仍需 digest、source authority 和 review-data identity 绑定。
- `[src:ai-proposed]` 不得自动变成已确认事实。
- 大型项目允许拆成多个导图，但必须保留一个可导航的全局入口与稳定节点身份。
- Level 1 必须以具体产品业务为主语；Constitution 只能提供跨项目治理约束和覆盖检查，不能成为业务能力图的事实来源。

## 密度合同

- 每张图最多 18 个可见节点、最多 3 层、每个父节点最多 4 个直接子节点。
- 节点数达到 8 个时，任一层不得承载超过 60% 的节点；不能用一层极稀疏、下一层堆满细节的方式规避拆图。
- 总图只放项目目标、核心角色、业务域、范围和全局约束入口；业务细节必须下钻到分图。
- 一个业务域超过单图预算时必须继续拆为子图，不能扩大预算或缩小字号硬塞。
- 横切政策、合规和治理规则只在全局约束图保留一个权威节点，通过 `affected_node_ids` 关联业务节点。
- 坐标、缩放和折叠仅属于浏览器本地状态；图归属、父子关系、下钻关系和影响范围属于语义合同。

## 错误记录

- 本机 Claude/Gemini 并行调用的编排层只返回终端能力警告，未返回正文；改为分别顺序调用并检查 JSON，不重复该并行方式。
