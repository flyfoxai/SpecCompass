# 审核页推荐项补齐与下载前检查设计

## 背景

审核页当前只在下载确认包前检查 `DRAFT` 草稿。对于从未选择的 `MISSING` 确认点，页面会直接生成确认包，因此新模块在尚未做任何选择时也能下载。现有“当前可见项按推荐保存”只处理当前节点或当前流程，无法一次补齐整个审核批次。

## 目标

- 下载确认包前检查整个审核批次中尚未选择的确认点。
- 允许用户确认后将所有可自动处理的未选择项按推荐保存，并继续下载。
- 提供全项目和当前视图两个明确的批量推荐入口。
- 任何批量操作都不得覆盖已有人工选择或草稿。

## 方案比较

### 方案 A：按状态补齐，保留已有选择（采用）

两个批量入口都只处理 `MISSING` 且存在 `recommended_option` 的确认点。`SAVED_RECOMMENDED`、`SAVED_SUBMITTED` 和 `DRAFT` 原样保留。下载入口使用同一套统计和补齐逻辑。

优点是不会覆盖人工决定，两个作用域清晰，下载检查与按钮行为一致。代价是草稿仍需用户单独处理。

### 方案 B：全量覆盖为推荐

“全部选择推荐”把所有确认点重写为推荐状态，包括非推荐提交和草稿。

操作最少，但会静默丢失人工审核结果，不采用。

### 方案 C：只阻止下载，不提供自动补齐

发现未选择项后仅提示用户返回逐项处理。

数据边界最严格，但新模块的批量初始化成本高，也不满足自动填写要求，不采用。

## 交互设计

右侧确认栏提供两个按钮：

- `全部选择推荐`：扫描所有模块、所有流程或界面，只将尚未选择且有推荐选项的确认点保存为推荐。
- `当前视图剩余项选推荐`：保留现有作用域规则。选中单个节点时只处理该节点；未选中节点时处理当前流程或界面的全部可见确认点。

点击任一按钮前，弹窗显示作用域、未完成数量、可自动填写数量，并说明已有选择和草稿不会被覆盖。用户取消时不改变状态；确认后保存推荐项并显示保存、跳过草稿、跳过已有选择和缺少推荐项的数量。

点击 `下载确认包` 时，先统计全项目状态：

1. 如果存在 `MISSING` 项，弹窗显示剩余数量和可按推荐补齐数量，询问是否将剩余未选项按推荐保存并继续下载。
2. 用户取消时不保存、不下载。
3. 用户确认后，仅补齐 `MISSING` 且有推荐选项的项目，然后继续原下载流程。
4. 如果仍存在缺少推荐选项的 `MISSING` 项，阻止下载并显示数量，避免生成看似完整但实际未完成的确认包。
5. 如果存在 `DRAFT`，沿用现有草稿警告和二次确认规则；自动补齐不得改变草稿。
6. 所有应确认项已解决后，正常生成确认包。

## 状态与数据边界

批量推荐的唯一可写前置条件为：

```text
requiresNodeDecision(node)
&& node.recommended_option
&& nodeState(node.id).status === "MISSING"
```

批量写入保持现有格式：

```json
{
  "status": "SAVED_RECOMMENDED",
  "option": "<recommended_option>"
}
```

下载检查覆盖 `allNodes()`，当前视图按钮覆盖 `visibleNodes()`。两个入口复用同一个统计与保存函数，避免作用域之外出现不同的状态判定。

## 异常处理

- 未加载 review data 时不执行批量保存或下载。
- 没有可保存项时不弹确认框，只显示当前作用域已处理完成。
- 缺少推荐选项的未选择项不能被伪造为已完成，也不能继续下载。
- 浏览器本地存储失败时沿用现有错误提示，不声称保存成功。
- 批量操作后重置下载按钮标签和导出指纹，确保新状态必须重新导出。

## 测试范围

- 新模块所有确认点均为 `MISSING` 时，首次下载必须弹出全项目补齐确认，不得直接下载。
- 确认补齐后，所有可推荐的 `MISSING` 项变为 `SAVED_RECOMMENDED`，并继续生成确认包。
- 取消下载确认时，状态和下载调用均不发生变化。
- `全部选择推荐` 跨模块处理全部未选择项。
- `当前视图剩余项选推荐` 只处理当前节点或当前流程。
- 两个批量入口都不覆盖 `DRAFT`、`SAVED_RECOMMENDED` 或 `SAVED_SUBMITTED`。
- 存在无推荐选项的 `MISSING` 项时阻止下载。
- 原有草稿二次确认、摘要复制和确认包内容测试继续通过。
